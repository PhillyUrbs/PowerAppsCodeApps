# Microsoft Roadmap Calendar — Build Specification

Use this document to build the complete app from scratch using any AI coding tool (GitHub Copilot, Claude Code, Cursor, etc.). Follow each section in order. The result should be a working Power Apps Code App that builds with zero errors.

---

## 1. What to Build

A single-page web app that aggregates features from the **M365** and **Azure** public roadmap APIs into a **12-month calendar grid** for a given year. Users can:
- Navigate forward/backward between years
- Filter by data source (M365, Azure), release ring, product, cloud instance, product category
- Search features by title or description
- Click a feature to see full details in a slide-out drawer
- See color-coded badges per ring type in each month cell
- Click a month name or "+N more" to open a **month detail panel** showing all features grouped by product
- Click a feature in the month detail panel to see its details, then return to the month panel when closing the detail drawer
- Collapse/expand the filter sidebar with a toggle button
- Filter options **cascade**: changing one filter updates the available options in other filters (e.g., selecting only M365 source hides Azure-only products)

## 5. Data Sources

### M365 Roadmap API
- **URL**: `https://www.microsoft.com/releasecommunications/api/v2/M365`
- **Dev proxy URL**: `/api/roadmap/M365` (use `import.meta.env.DEV` to switch)
- **Pagination**: OData `$skip` query param, 100 items per page; follow `@odata.nextLink` until no more
- **Response shape** (each item in `value[]`):
```ts
{
  id: number,
  title: string,
  description: string,           // may contain HTML
  cloudInstances: string[],      // e.g. ["Worldwide (Standard Multi-Tenant)", "GCC", "GCC High", "DoD"]
  platforms: string[],           // e.g. ["Desktop", "Web", "iOS", "Android", "Mac"]
  releaseRings: string[],        // e.g. ["General Availability", "Targeted Release", "Preview"]
  moreInfoUrls: string[],
  products: string[],            // e.g. ["Microsoft Teams", "Outlook", "SharePoint"]
  generalAvailabilityDate: string | null,   // "YYYY-MM" format
  previewAvailabilityDate: string | null,
  created: string,               // ISO datetime
  status: string,                // "Launched" | "Rolling out" | "In development" | "Cancelled"
  modified: string,
  availabilities: Array<{ ring: string, year: number, month: string }>
  // month is the English name: "January", "February", etc.
}
```

### Azure Roadmap API
- **URL**: `https://www.microsoft.com/releasecommunications/api/v2/Azure`
- **Dev proxy URL**: `/api/roadmap/Azure`
- **Pagination**: Same OData `$skip` pattern
- **Response shape** (each item in `value[]`):
```ts
{
  id: string,                    // string ID, not number
  title: string,
  description: string,           // contains HTML
  productCategories: string[],   // e.g. ["Compute", "Networking", "Databases", "Security"]
  tags: string[],                // e.g. ["Features", "Microsoft Ignite", "Retirements"]
  products: string[],            // e.g. ["Azure Kubernetes Service (AKS)", "Azure SQL Database"]
  generalAvailabilityDate: string | null,
  previewAvailabilityDate: string | null,
  privatePreviewAvailabilityDate: string | null,  // Azure-only field
  status: string | null,         // "Launched" | "In preview" | "In development" | null (retirements)
  created: string,
  modified: string,
  availabilities: Array<{ ring: string, year: number, month: string }>
  // ring values include: "General Availability", "Preview", "Private Preview", "Retirement"
}
```

### Unified Type

Both APIs must be normalized into a single type:
```ts
interface RoadmapFeature {
  id: string;                    // prefix with "m365-" or "azure-" to avoid collisions
  title: string;
  description: string;
  source: 'M365' | 'Azure';
  status: string;
  products: string[];
  platforms: string[];           // M365 only, empty for Azure
  cloudInstances: string[];      // M365 only, empty for Azure
  productCategories: string[];   // Azure only, empty for M365
  tags: string[];                // Azure only, empty for M365
  releaseRings: string[];
  generalAvailabilityDate: string | null;
  previewAvailabilityDate: string | null;
  privatePreviewAvailabilityDate: string | null;  // Azure only, null for M365
  availabilities: Array<{ ring: string; year: number; month: string }>;
  created: string;
  modified: string;
  moreInfoUrls: string[];
}
```

## 6. Core Logic

### CORS / Dev Proxy
- The Microsoft roadmap APIs block CORS from localhost. Use a Vite dev server proxy (see vite.config.ts above).
- In API fetcher files, use `import.meta.env.DEV` to switch between `/api/roadmap/M365` (dev) and the full URL (production).

### Fetching
- Fetch both APIs in parallel (`Promise.all`)
- Each fetcher paginates using `$skip=0`, `$skip=100`, etc. until `@odata.nextLink` is absent or page returns fewer than 100 items
- Map each raw API item to the unified `RoadmapFeature` type
- Use TanStack Query with `queryKey: ['roadmap-features']`

### Grouping by Month
- For each feature, iterate its `availabilities[]` array
- Each availability has `{ ring, year, month }` where month is the English name (e.g. "March")
- A single feature can appear in **multiple months** (e.g., Preview in March and GA in June)
- Group into a `Map<monthNumber, MonthFeature[]>` where `MonthFeature = { feature, ring }`
- Initialize all 12 months (1-12) with empty arrays

### Filtering
Apply filters before grouping. Filter criteria:
- **Source**: M365, Azure (checkbox toggles, both on by default)
- **Rings**: filter by ring names from `availabilities[].ring` (empty = show all)
- **Products**: match against `feature.products[]`
- **Cloud Instances** (M365-only): match against `feature.cloudInstances[]`
- **Product Categories** (Azure-only): match against `feature.productCategories[]`
- **Text search**: case-insensitive substring match on `title` or `description`

### Cascading Filter Options (IMPORTANT)
Filter options must cascade — each filter dimension shows only values that are reachable given the current selections in OTHER filters:
1. First, pre-filter features by the selected sources (M365/Azure).
2. For each filter dimension (products, rings, cloudInstances, productCategories), apply all OTHER active filters to determine which features are reachable, then extract that dimension's values from the reachable features.
3. This means: if only "M365" source is selected, only M365 products appear in the Products list. If a specific ring is selected, only products that have features in that ring appear.
4. Implement a `matchesFiltersExcept(feature, filters, excludeDimension)` helper that checks all filter criteria except one dimension.

### Product List Sorting
In the products filter list, sort with **selected (checked) items first**, then alphabetically within each group. Show all matching products (no arbitrary cap). Include a sub-search input for filtering the product list.

## 7. UI Components

### Layout (App.tsx)
- Full viewport height, flexbox column
- **Header**: app title "Microsoft Roadmap Calendar" on left, version `v1.0.0` on right
- **Body**: horizontal flex — collapsible filter sidebar on left, scrollable main area on right
- **Main area**: YearNavigation at top, stats bar showing total/filtered counts, then CalendarGrid
- **Detail drawer**: `OverlayDrawer` from Fluent UI, slides from right when a feature is clicked
- **Month detail panel**: fixed-position panel, centered horizontally, opens when a month is clicked
- **State**: `year`, `filters`, `selectedFeature`, `detailOpen`, `expandedMonth`

### YearNavigation
- Centered row: left chevron button, year as `Title1`, right chevron button
- Clicking buttons increments/decrements the year state

### CalendarGrid
- CSS Grid: `grid-template-columns: repeat(auto-fill, minmax(280px, 1fr))`
- Renders 12 `MonthCell` components (January through December)
- Passes `onMonthClick` callback for month detail view

### MonthCell
- Fluent UI `Card` component
- Header: **clickable** month name (opens month detail) + ring count badges (small colored `Badge` components)
- Body: list of up to 8 `FeatureCard` items, then **clickable** "+N more" text (also opens month detail)
- Empty state: italic "No features this month"

### FeatureCard
- Single row: colored dot (ring color) + truncated title + source badge ("M365" or "Azure")
- Clickable (role="button"), keyboard accessible
- Compact: 24px min height

### MonthDetailDialog (IMPORTANT — custom panel, NOT Fluent Dialog)
This is a **plain positioned `div`**, not a Fluent UI Dialog (Dialog caused visual glitches with non-modal mode).
- **Position**: `position: fixed`, horizontally centered (`left: 50%, transform: translateX(-50%)`), `top: 5vh`, `bottom: 5vh`
- **Width**: `720px`, `maxWidth: calc(100vw - 300px)`
- **Styling**: uses Fluent tokens for background (`colorNeutralBackground1`), border (`colorNeutralStroke1`), border-radius (`borderRadiusXLarge`), shadow (`shadow64`), `zIndex: 1000`
- **Header**: fixed at top with month name, year, feature count, and dismiss button
- **Content**: scrollable area, features **grouped by primary product** (first product in `products[]`), sorted alphabetically by product then title
- **Each feature row**: ring color dot, full title (wrapping, NOT truncated), product list below title, source badge
- **Close behavior**:
  - Dismiss button closes it
  - Clicking outside the panel closes it (use `mousedown` event listener on `document`, added via `requestAnimationFrame` to avoid immediate trigger)
  - Escape key closes it
  - **Exception**: when the feature detail drawer is open (`detailDrawerOpen` prop), the click-outside handler is **disabled** so clicking the drawer backdrop doesn't close the month panel
- **Interaction with detail drawer**: clicking a feature opens the detail drawer WITHOUT closing the month panel. When the drawer is closed, the month panel is still visible.

### FeatureDetailPanel
- Fluent UI `OverlayDrawer` (position="end", size="medium")
- Shows: title, ring badge, source badge, status badge
- Divider, then description text (strip HTML tags using DOMParser)
- Sections for: Products, Timeline (from availabilities), Cloud Instances, Platforms, Categories
- Each section only renders if the feature has data for it

### FilterPanel (Collapsible)
- **Collapsed state**: narrow 40px strip with a filter icon button; click to expand
- **Expanded state**: 260px width sidebar with border-right
- **Transition**: `width` transition `0.2s ease`
- **Header**: "Filters" title + collapse/expand toggle button (use `FilterRegular`/`FilterDismissRegular` icons)
- Search input at top with search icon
- Accordion sections (Fluent UI `Accordion`):
  - **Source**: M365/Azure checkboxes
  - **Ring / Status**: checkboxes with colored dots matching ring colors
  - **Products (N)**: sub-search input + scrollable checkbox list (max-height 300px), count in header
  - **Cloud Instance (M365)**: checkboxes, label indicates M365-only (only renders if values exist)
  - **Category (Azure)**: scrollable checkbox list with max-height 200px, label indicates Azure-only (only renders if values exist)
- Source and Rings sections open by default (`defaultOpenItems={['source', 'rings']}`)

## 8. Entry Point (main.tsx)

```tsx
import { StrictMode } from 'react';
import { createRoot } from 'react-dom/client';
import { QueryClient, QueryClientProvider } from '@tanstack/react-query';
import { FluentProvider, webDarkTheme } from '@fluentui/react-components';
import App from './App.tsx';
import './index.css';

const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 30 * 60 * 1000,
      refetchOnWindowFocus: false,
    },
  },
});

createRoot(document.getElementById('root')!).render(
  <StrictMode>
    <QueryClientProvider client={queryClient}>
      <FluentProvider theme={webDarkTheme}>
        <App />
      </FluentProvider>
    </QueryClientProvider>
  </StrictMode>,
);
```

## 10. Build Verification

After generating all files, run:
1. `npm install`
2. `npm run build` (runs `tsc -b && vite build`)

The build must succeed with **zero TypeScript errors** and produce a `dist/` folder with `index.html` and assets.

**Common pitfalls:**
- Unused imports cause TS errors (strict mode). Remove any unused imports.
- Fluent UI `Checkbox` component does not accept `size="small"` — use `size="medium"` or omit.
- `@tanstack/react-query` must install with its `build/` directory — if types are missing, reinstall at a pinned version.
- Descriptions from the APIs contain HTML. Strip tags with `new DOMParser().parseFromString(html, 'text/html').body.textContent`.
- Do NOT use Fluent UI `Dialog` for the month detail panel — it causes visual glitches in non-modal mode. Use a plain positioned `div` instead.

## 11. Known Limitations

- **CORS**: The M365 and Azure roadmap APIs block CORS from localhost. The Vite dev proxy handles this for local development. For production (Power Apps sandbox), you may need an Azure Function proxy.
- **Data volume**: Both APIs return hundreds to thousands of features. The app fetches all pages on load. Consider lazy loading per-year or adding API-side `$filter` if available.
- **Release Plans**: A third data source (`releaseplans.microsoft.com` for Power Platform/Dynamics 365) was investigated but no public API was found. The site is a JavaScript SPA with no exposed REST endpoints. This is deferred to v2.
